# アプリケーション企画設計書：「Pizack（ピザック）」

## 1. プロジェクト概要

### コンセプト

**「タスクを食らえ！ 成果を積み上げろ！」** 開発タスクの消化を「ピザ作り」になぞらえ、個人の達成感とチームの進捗を直感的に可視化する、学生・ハッカソンチーム向け開発支援ツール。

### 背景・解決する課題

情報系学生のチーム開発において、既存のビジネスライクなツール（Trello, Jira等）では以下の問題が発生しやすい。

- 無機質さ: 事務的な作業になりがちで、モチベーションが続かない。
- タスクの巨大化: 初心者はタスクを大きく定義しがちで、手が止まる（着手への心理的ハードルが高い）。
- 進捗の不透明さ: 誰がどれくらい貢献しているかが見えにくく、チームの一体感が損なわれる。

**本アプリは、ゲーミフィケーションとアジャイル開発のベストプラクティス（タスク細分化）を融合させることで、これらの課題を解決する。**

## 2. ユーザー体験 (UX) の核となるメタファー

開発プロセスをピザ屋の厨房に置き換える。

| 開発用語 | アプリ内のメタファー | 役割・効果 |
| ---- | ---- | ---- |
| 機能要件 (親タスク) | オーダー (ピザ生地) | 1つの機能＝1枚のピザとして管理。 |
| 作業工程 (子タスク) | トッピング (具材) | タスクを分解するほど具材が増える。 |
| 完了 (Merge/Done) | 箱詰め (Delivery) | 成果物としてチームの共有スペースに積まれる。 |
| バグ修正 | Hotfix Pizza | 焼き直しはしない。修正用ピザを新たに焼いて積む。 |

## 3. 機能要件詳細

### A. My Kitchen（個人作業画面）

ユーザーが自分のタスクに集中するための「厨房」画面。

1. オーダー管理（タスク登録）
  - マルチタスク対応: 「まな板」は複数用意され、並行して複数の機能開発（ピザ作り）を進めることができる。
  - 保留機能: 手が空かないタスクは「冷蔵庫」リストに退避させ、まな板を整理できる。

2. レシピ作成（タスク分解の誘導）
  - 親タスク（生地）に対し、子タスク（具材）を追加する。
  - 【重要】ピザグレードシステム: 子タスクの数に応じて、完成時の「箱の色」が変化する。
    - 1〜2個 → 緑の箱 (Snack Box): 雑務、修正など。
    - 3〜5個 → 赤の箱 (Regular Box): 標準的な機能実装。
    - 6個以上 → 金の箱 (Deluxe Box): 詳細に分解された、重要機能の実装。
  - 狙い: 「金の箱を積み上げたい」という欲求を利用し、タスクを細かく分解（＝着手しやすくする）行動を自然に促す。

3. 調理プロセス（可視化）
  - 子タスクのチェックボックスを埋めるごとに、ピザの画像がリアルタイムで変化する。
  - レイヤー合成技術: 生地に「ソース」「チーズ」「具材」の画像をプログラム上で重ね合わせることで、進捗率を直感的に表現する。
  - ロジック: 完了数 / 全タスク数 の割合に応じて具材を描画。

4. 完了アクション
  - 全ての子タスクが完了すると「焼く」ボタンが有効化。
  - 効果音と共にピザが箱詰めされ、アニメーションで画面端の「自分のタワー」に追加される。

### B. Delivery Counter（チーム共有画面）

チームメンバー全員の成果を一望できる画面。

1. ピザタワーの表示
  - メンバーごとに横並びで「ピザの箱」が下から上へ積み上がる。
  - 箱の色（緑・赤・金）により、単なる枚数だけでなく「仕事の質・重さ」も可視化される。

2. 非同期コミュニケーション
  - リアルタイム監視ではなく、ページを開いたタイミング（または定期更新）で最新のタワー状態を表示。
  - 「A君のタワーが高いな（負けられない）」「B君、まだ箱がないな（詰まってるかも？）」といった状況把握を、テキストチャットなしで実現する。

3. 詳細情報の表示（タイムスタンプ）
  - 各ピザの箱にはメタデータとして「完了日時」「担当者」「タスク名」を保持する
  - UIのシンプルさを保つため、常時表示はせず、マウスオーバー（ホバー）時のみツールチップで詳細情報を表示する。

### C. History（振り返り画面）

- プロジェクト終了後のリザルト画面。
- 総出荷枚数（Total Pizza）の表示。
- MVP（最も多くのタスクを消化したメンバー）の表彰など、チームの労をねぎらう演出。

## 4. 技術スタック・運用設計

### Frontend

- Core: React (Vite) + TypeScript
  - 選定理由: 開発市場の標準技術であり、TypeScriptの静的型付けによってバグを未然に防ぎ、チーム開発におけるコードの品質と可読性を担保するため。
- Styling: Tailwind CSS
  - 選定理由: クラス名ベースで素早くスタイリングを行い、UI構築の試行錯誤（プロトタイピング）を高速化するため。

### Backend / Database

- Platform: Firebase (Firestore / Authentication)
  - 選定理由 1 (機能面): アプリの核となる「チーム間の進捗共有（ピザタワー）」において、Firestoreのリアルタイムリスナー機能が不可欠であり、WebSocket等を自前で構築する工数を削減するため。
  - 選定理由 2 (コスト面): MVP（実用最小限の製品）段階では、Sparkプラン（無料枠）の範囲内で十分に運用可能であり、学生プロジェクトとしてランニングコストをゼロに抑えるため。将来的なスケーリングが必要になった段階で、構成の見直しを行う。

### Platform / Infrastructure

- Type: Webアプリケーション (Single Page Application)

  - 運用方針: PCブラウザでの利用を主とするが、外出先からの進捗確認も考慮し、スマートフォンでもレイアウトが崩れないレスポンシブ設計とする。

- Hosting: Firebase Hosting (または Vercel)

- CI/CD: GitHub Actions (将来的な自動デプロイを見据える)

## 5. 開発スコープ（MVP）

### Phase 1: ソロプレイモード（個人開発）

- Reactのみで実装。
- タスク登録、画像の変化、ローカルでの箱積み上げまでを完成させる。
- 目的: 基本的なUI/UXの構築と、「ピザ作り」の楽しさの検証。

### Phase 2: チームプレイモード（クラウド連携）

- Firebaseを導入。
- チームメンバー間でのデータ共有（Delivery Counter）の実装。
- 目的: 「チーム開発支援ツール」としての価値の証明。
